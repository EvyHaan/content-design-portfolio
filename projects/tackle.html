WORK HELLO

Web form component review & recommendations for Tackle.io

This component review was a self-driven exercise in exploring UX Writing "in the wild".


Project Type

Component review & proposed recommendations


Timeline

Mar 2022


Role

UX Writer


Tools

Google Docs/Sheets/Slides, diagrams.net, Firefox & Chrome browser inspect element tools, WAVE accessibility evaluation tool

Overview


While reviewing the website for Tackle.io, I noticed the CTA for the newsletter sign-up form had text spilling onto a second line. Looking a little closer, I noticed other ways I might suggest to improve that form. As I scanned through the rest of the website I noticed two other forms, each with a slightly different design.

In this redesign proposal, my goals are to:

  1. Continuity across the website forms
  2. Improve error copy
  3. Pose considerations for accessibility

These are the forms I address:

  • Newsletter Sign-up
  • Sign in with email
  • Schedule a demo

Understand


Below are the current forms.

Observations

Design.

Thinking about continuity, I used a spreadsheet to compare the component elements across the forms.

I also logged all the error messages. Most messages are generic and do not give the user helpful context (“Please complete this required field”). The copy could be improved to sound a little less mechanical.

Key takeaways:

  • The float labels seem to be a good solution for including labels/placeholder text.
  • The “Sign in with email” CTA needs to be style-matched to the rest of the website.
  • The “Newsletter sign up” CTA text needs to be on one line.
  • Sentence casing needs to be applied consistently across each component element.

Development.

Using the inspector tool in the browser, I can see a little of how the designs have been programmatically implemented.

Specifically, I wanted to understand if there was a programmatic reason the CTA text on the Newsletter Sign Up form was on two lines because it seemed so close but slightly inconsistent to the design.

I learned that while other CTA text had a set font-size: 14px, this CTA had a responsive font-size: 1em. In most browsers, 1em is equivalent to 16px.


Resizing the font to 14px brought the text onto one line.

Key takeaway:

I understood this to mean there may be opportunity for cross-collaboration between design and development teams to discuss and mutually define dimension values.


Accessibility.

I had some immediate thoughts around accessibility when first viewing the forms:

  • Two of the forms do not have labels
  • One of the forms does not have placeholder, hint, or instruction text
  • One form has placeholder text that disappears when the input field is focused

I used the WAVE accessibility evaluation extension for more accessibility insights relevant to content.

Notes from the Newsletter Sign Up form:

  • The instruction text above the form also does not have great contrast
  • There are a few noscript tags, which may be hiding content from assistive technology like screen readers

Notes from the Schedule a Demo form:

  • Labels are well-used in most instances
  • Again, there are a few noscript tags, potentially hiding content

Notes from the Sign In with Email form:

  • Labels are not present
  • Confirmed low contrast of “Forget password?” text

I found these findings were generally framed around visual impairment and assistive technology. I had questions about other potential users.


What if the user is neurodivergent? They may not have visual impairments, but could still benefit from constant labels and placeholders.


Or perhaps a user is generally not very tech savvy? Or they use the keyboard tab to quickly switch fields and don’t have a chance to see placeholder text before it disappears. These forms are relatively basic. But thinking big-picture, forms may be more complex elsewhere on the site and require more careful crafting of labels, placeholder text, and instruction copy. This ties back into the idea of sitewide consistency.

Recommendations


Goal 1 - Consistency across the website forms

After an in-depth review of these different fields, a few recommendations to improve consistency are:

  • Include labels for all form fields visible above their input fields
  • Visible labels provides a natural contextual space if any additional instruction needs to be provided for a user to complete that field in lieu of placeholder text
  • If placeholder text is used, it should have consistent casing
  • Convert font-size dimensions from pixels to ems for resizability by users

Goal 2 - Improve error copy

I revised the error messages that I had logged. Knowing the users of the site are business-minded, I wanted the voice to come across as professional and succinct. The tone needed to be direct, but a step short of being mechanical. And I aimed to include some context to each error. These are my suggestions.


There were two particularly challenging errors:


Incorrect format - phone number. This field is challenging on three levels. First, it should allow for international phone numbers. Second, it should allow for extensions. Third, it should not be over-restrictive on how users input their phone numbers. However, given the current error message, I could use the allowable characters and still receive an error message. I aimed to add a little more specificity in how to use certain special characters.


Correct email format, incorrect password. Rather than stating when an email or a password is invalid or incorrectly formatted, the error waits to validate both. The payoff of error specificity for improved UX versus security measures is still in debate. Because the form is small enough, I am ok with the ambiguous error message with a few caveats:


  • The wording be improved to not tell the user they are “wrong”
  • An additional error message if the email format is incorrect
  • Additional error messages if the email or password fields are left blank

Goal 3 - Accessibility considerations

I really only touched on surface-level accessibility uses. Many of the recommendations around what I observed would be addressed by the recommendations around consistency across all website forms. The only additional point would be addressing any color contrast issues.

Reflection


I was motivated to take a deeper look into the UX writing aspects of the forms on Tackle’s website because of the two-lined “Sign Up” CTA on the Newsletter Sign Up form. I could see what was intended by design, but I could also see what potentially was mis-implemented in the development side. Once I dug into the development side of things, it started a chain of questions around accessibility and usability.


That is what compelled me to write a mini case study about these forms. The most important thing I gained from this exercise were the questions I had at the intersections of UX writing, accessibility, and development. Great UX isn’t achieved by any one of these teams alone, but when these teams collaborate.


Even the recommendations I pose here, still feel tentative without having those questions answered. There is still room for continual iteration and improvement. There is much more to research. There is still more to consider.
